## Core Capabilities

### 1. TypeORM Entity Design

All entities follow these rules without exception:

```typescript
import {
  Entity,
  PrimaryGeneratedColumn,
  Column,
  CreateDateColumn,
  UpdateDateColumn,
  DeleteDateColumn,
  Index,
  ManyToOne,
  JoinColumn,
} from 'typeorm';

@Entity('kms_files')
@Index(['sourceId', 'createdAt'])
@Index(['status', 'createdAt'])
export class KmsFile {
  @PrimaryGeneratedColumn('uuid')
  id: string;

  @Column({ name: 'source_id', type: 'uuid' })
  @Index()
  sourceId: string;

  // Soft reference to voice_jobs — NOT a FK constraint
  @Column({ name: 'voice_job_id', type: 'uuid', nullable: true })
  voiceJobId: string | null;

  @Column({ type: 'varchar', length: 500 })
  originalName: string;

  @Column({ name: 'minio_object_key', type: 'varchar', length: 1000 })
  minioObjectKey: string;

  @Column({ name: 'mime_type', type: 'varchar', length: 255 })
  mimeType: string;

  @Column({ name: 'file_size_bytes', type: 'bigint' })
  fileSizeBytes: number;

  @Column({
    type: 'enum',
    enum: ['pending', 'scanning', 'embedding', 'deduping', 'completed', 'failed'],
    default: 'pending',
  })
  status: string;

  @Column({ name: 'sha256_hash', type: 'varchar', length: 64, nullable: true })
  @Index({ unique: true, where: 'sha256_hash IS NOT NULL' })
  sha256Hash: string | null;

  @Column({ name: 'is_junk', type: 'boolean', default: false })
  isJunk: boolean;

  @Column({ name: 'junk_score', type: 'float', nullable: true })
  junkScore: number | null;

  @Column({ name: 'duplicate_cluster_id', type: 'uuid', nullable: true })
  duplicateClusterId: string | null;

  @Column({ name: 'error_message', type: 'text', nullable: true })
  errorMessage: string | null;

  @CreateDateColumn({ name: 'created_at' })
  createdAt: Date;

  @UpdateDateColumn({ name: 'updated_at' })
  updatedAt: Date;

  @DeleteDateColumn({ name: 'deleted_at' })
  deletedAt: Date | null;
}
```

**Rules:**
- All PKs are UUIDs generated by PostgreSQL (`gen_random_uuid()`), not application-generated.
- All column names are snake_case in the database (`name: 'column_name'`), camelCase in TypeScript.
- Timestamps: `created_at`, `updated_at` always present. `deleted_at` for soft-delete tables.
- Enum columns use PostgreSQL native `ENUM` type.
- Cross-domain references are plain `uuid` columns — no `@ForeignKey` or `@ManyToOne` across table namespaces.

### 2. Migration Strategy

Every schema change requires a TypeORM migration file. No exceptions.

```typescript
// migrations/{timestamp}-AddSha256HashToKmsFiles.ts
import { MigrationInterface, QueryRunner, TableIndex, TableColumn } from 'typeorm';

export class AddSha256HashToKmsFiles1705312260000 implements MigrationInterface {
  name = 'AddSha256HashToKmsFiles1705312260000';

  public async up(queryRunner: QueryRunner): Promise<void> {
    await queryRunner.addColumn(
      'kms_files',
      new TableColumn({
        name: 'sha256_hash',
        type: 'varchar',
        length: '64',
        isNullable: true,
      }),
    );

    await queryRunner.createIndex(
      'kms_files',
      new TableIndex({
        name: 'UQ_kms_files_sha256_hash',
        columnNames: ['sha256_hash'],
        isUnique: true,
        where: 'sha256_hash IS NOT NULL',
      }),
    );
  }

  public async down(queryRunner: QueryRunner): Promise<void> {
    await queryRunner.dropIndex('kms_files', 'UQ_kms_files_sha256_hash');
    await queryRunner.dropColumn('kms_files', 'sha256_hash');
  }
}
```

Migration file naming: `{timestamp}-{PascalCaseDescription}.ts`
- `timestamp` = Unix milliseconds at time of creation
- Always implement `down()`. A migration without `down()` is rejected.
- Never modify an existing migration file after it has been applied to any environment.
- To fix a migration error, create a new corrective migration.

### 3. Index Strategy

| Index Type  | Use Case                                            | Column Types                        |
|-------------|-----------------------------------------------------|-------------------------------------|
| B-tree      | Range queries, ORDER BY, equality on timestamps/UUIDs | `created_at`, `updated_at`, `id`    |
| Hash        | Exact equality only (no range), faster than B-tree for equality | `sha256_hash`, `api_key_hash`     |
| GIN         | Full-text search (`tsvector`), JSONB containment    | `content_tsv`, `metadata`           |
| GIN trigram | ILIKE pattern matching, fuzzy name search           | `original_name`, `source_name`      |
| Partial     | Filtered queries on a subset of rows                | `status` where `status != 'completed'` |
| Composite   | Multi-column lookups (order matters)                | `(source_id, created_at)`           |

```sql
-- Full-text search index
CREATE INDEX CONCURRENTLY idx_kms_file_chunks_content_fts
ON kms_file_chunks USING GIN(to_tsvector('english', content));

-- Trigram index for name search
CREATE EXTENSION IF NOT EXISTS pg_trgm;
CREATE INDEX CONCURRENTLY idx_kms_files_name_trgm
ON kms_files USING GIN(original_name gin_trgm_ops);

-- Hash index for SHA-256 lookup
CREATE INDEX CONCURRENTLY idx_kms_files_sha256_hash
ON kms_files USING HASH(sha256_hash)
WHERE sha256_hash IS NOT NULL;

-- Composite index for source + time range queries
CREATE INDEX CONCURRENTLY idx_kms_files_source_created
ON kms_files(source_id, created_at DESC);
```

Use `CREATE INDEX CONCURRENTLY` in migrations for large tables to avoid table locks.

### 4. Cross-Domain Isolation Rule

**Never add a foreign key constraint between table namespaces.**

- `kms_*` tables reference each other freely with FK constraints.
- `voice_*` tables reference each other freely with FK constraints.
- `auth_*` tables are referenced by `kms_*` tables (API key → files is allowed).
- Cross-namespace references (kms → voice, voice → kms) are stored as plain UUID columns.

Rationale: Each namespace is owned by a different service. Foreign key constraints would create deployment ordering dependencies and prevent independent schema evolution.

When a `kms_file` is deleted, the application layer must clean up the `voice_jobs` reference separately via a soft reference lookup — not via cascade.

### 5. Qdrant Collection Design

**Collection: `kms_files_default`**
- Embedding model: `sentence-transformers/all-MiniLM-L6-v2` (384 dimensions)
- Distance metric: `Cosine`
- Point ID: must match `kms_file_chunks.id` (UUID)
- Payload: `{ fileId, sourceId, chunkIndex, mimeType, createdAt }`

**Collection: `kms_files_cloud`**
- Embedding model: `text-embedding-3-small` (1536 dimensions) — OpenAI
- Distance metric: `Cosine`
- Point ID: must match `kms_file_chunks.id` (UUID)
- Payload: same as default

```python
from qdrant_client import QdrantClient
from qdrant_client.models import VectorParams, Distance

client = QdrantClient(url=settings.QDRANT_URL)

client.create_collection(
    collection_name="kms_files_default",
    vectors_config=VectorParams(size=384, distance=Distance.COSINE),
    on_disk_payload=True,  # keep payload on disk for large payloads
)
```

**Index fields in Qdrant payload** (for filtered search):
- `fileId` — keyword index (exact match filter)
- `sourceId` — keyword index (scoped search by source)
- `mimeType` — keyword index (filter by file type)
- `createdAt` — integer index (range filter by date)

### 6. Neo4j Graph Schema

**Node labels:**
- `File` — properties: `id` (UUID, indexed), `sha256Hash`, `name`, `mimeType`, `createdAt`
- `DuplicateCluster` — properties: `id` (UUID), `createdAt`

**Relationship types:**
- `(File)-[:DUPLICATE_OF {similarity: float, method: string}]->(File)`
- `(File)-[:BELONGS_TO_CLUSTER]->(DuplicateCluster)`

**Cypher constraints:**
```cypher
CREATE CONSTRAINT file_id_unique IF NOT EXISTS
FOR (f:File) REQUIRE f.id IS UNIQUE;

CREATE CONSTRAINT cluster_id_unique IF NOT EXISTS
FOR (c:DuplicateCluster) REQUIRE c.id IS UNIQUE;

CREATE INDEX file_sha256_index IF NOT EXISTS
FOR (f:File) ON (f.sha256Hash);
```

When deleting a file from the KMS:
```cypher
MATCH (f:File {id: $fileId})
DETACH DELETE f
```
`DETACH DELETE` removes all relationships. Run this before deleting the PostgreSQL row.

### 7. Redis Caching Patterns

**Key naming convention:** `{namespace}:{entity}:{identifier}:{variant}`

```
search:results:{query_hash}          → search result set, TTL: 300s
search:similar:{fileId}              → similar files, TTL: 300s
session:{userId}                     → session data, TTL: 3600s
ratelimit:{apiKeyId}:{window}        → rate limit counter, TTL: 60s
```

**Cache invalidation:**
- When a file is deleted: `DEL search:similar:{fileId}` and invalidate search result pages containing that file (prefix scan: `SCAN 0 MATCH search:results:* COUNT 100`, then filter).
- When a source is deleted: flush all search result caches for that source.
- Do not cache search results that include deleted or junk files. Apply filters before caching.

**Eviction policy:** `allkeys-lru`. Redis must be configured with a memory limit. Do not run Redis without a `maxmemory` setting.

## Query Optimization Checklist

Before any query goes to production:

- [ ] Run `EXPLAIN ANALYZE` and verify the planner uses the expected index.
- [ ] No sequential scans on tables with more than 10,000 rows.
- [ ] No `SELECT *` in application queries — enumerate columns.
- [ ] Pagination queries use cursor-based approach, not `OFFSET`.
- [ ] Full-text search uses `tsvector` + GIN index, not `ILIKE '%term%'`.
- [ ] N+1 query patterns detected and resolved with JOIN or DataLoader.
- [ ] Long-running queries have a `statement_timeout` set at the session level.
- [ ] Write queries use `SELECT ... FOR UPDATE` when update depends on current state.

## Transaction Safety Rules

1. Use transactions for any operation that writes to more than one table.
2. Always acquire locks in a consistent order to prevent deadlocks (e.g., always lock `kms_sources` before `kms_files`).
3. Never hold a transaction open while performing external I/O (MinIO, RabbitMQ calls).
4. Use `SERIALIZABLE` isolation only when strict consistency is required; default to `READ COMMITTED`.
5. Implement retry logic for `serialization failure` (error code `40001`) with exponential backoff.

## FK-Safe Deletion Order for KMS Tables

When deleting a source or file, follow this order to avoid FK violations:

**Delete a file:**
1. Delete Qdrant points for all `kms_file_chunks` of this file
2. `DETACH DELETE` Neo4j `File` node
3. Delete `voice_jobs` records referencing this file (soft reference lookup, no FK cascade)
4. Delete `kms_file_chunks` (FK child of `kms_files`)
5. Soft-delete `kms_files` (`deleted_at = NOW()`)
6. Delete MinIO object

**Delete a source:**
1. For each file in the source, execute the file deletion order above
2. Soft-delete `kms_sources`

Never reverse this order. Never delete the parent before children.
