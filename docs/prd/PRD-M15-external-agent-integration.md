- **Status**: Draft

- **Author**: Claude Code (claude-sonnet-4-6)

- **Updated**: 2026-03-17
- **Depends on**: PRD-M13 (ACP Integration), PRD-M14 (Agentic Workflows)

## 1. Business Context

KMS M13 established KMS as an ACP *server* — a knowledge agent that external clients can query. M14 built the internal Workflow Engine that orchestrates multi-step pipelines using KMS-native agents. But KMS's LLM generation tier remains local-only (Ollama / OpenRouter): powerful in reasoning over retrieved context, but constrained by model quality compared to frontier models like Claude or Gemini.

M15 makes KMS **bidirectional**: KMS can now also act as an ACP *client* that delegates generation to external frontier agents, while remaining the exclusive owner of retrieval, context packaging, and citation tracking. The value proposition is clear: KMS knows your private documents in depth; Claude or Gemini knows how to reason and generate with world-class quality. Neither alone is as effective as both combined — KMS builds the richest possible retrieval context, external agents do the generation with that context.

Without M15, a user who wants Claude-quality answers grounded in their private knowledge base must manually copy-paste document excerpts into the Claude UI, losing citations, losing graph context, and losing any audit trail. With M15, that workflow is a single authenticated API call: the RAG pipeline retrieves, packs context, routes to Claude, and relays a citation-annotated streamed response — with every token counted and every source attributed.

M15 also closes the inverse gap: Claude Code and other IDE-embedded agents currently cannot query a user's private KMS knowledge base without custom scripting. M15 exposes KMS tools via the Model Context Protocol (MCP), letting Claude Code discover and call `kms_search`, `kms_retrieve`, and `kms_graph_expand` autonomously during a coding session. The knowledge base becomes a live, authenticated tool available to any MCP-compatible agent.

## 3. Scope

### In Scope

- **External Agent Registry** — configure, list, health-check, and test external agents (Claude Code via stdio ACP, Claude API via HTTP, Codex, Gemini, Kimi, custom ACP endpoints)
- **RAG → External Agent Context Pipeline** — Pattern 1: KMS retrieves context, Context Packer formats it, ACP Client sends to external agent, KMS relays SSE to user
- **Context Packer** (`rag-service/app/services/context_packer.py`) — formats RAG retrieval results into ACP resource blocks with token budget management, citation injection, and graph context inclusion
- **Stdio subprocess adapter** — spawns and manages `claude-agent-acp`, `codex-acp`, `gemini acp`, `kimi acp` child processes; handles process lifecycle
- **HTTP adapter** — connects to Anthropic API directly, or any ACP-over-HTTP endpoint
- **MCP Server** (`kms-api/src/modules/mcp/`) — Pattern 2: exposes `kms_search`, `kms_retrieve`, `kms_graph_expand` as MCP tools for Claude Code and other MCP clients; JWT-authenticated
- **`external_agent` workflow step type** — Pattern 3: WorkflowEngine step that delegates to any configured external agent with prior-step context
- **Two new tools**: `kms_ask_agent` and `kms_delegate` registered in the ACP tool registry
- **Streaming response relay** — SSE from external agent proxied back to KMS client with citation annotations
- **Context budget management** — per-agent token limits enforced at Context Packer; hard truncation by default, configurable summarisation mode
- **Audit logging** — extend `kms_acp_tool_calls` to record `external_agent_id`, `context_tokens_sent`, `response_tokens`, `agent_model`
- **Feature flags** under `externalAgents.*` and `mcp.*` — all new behaviour disabled by default
- **Error code domain KBEXT0001–KBEXT0012**

### Out of Scope

- Building, fine-tuning, or hosting custom LLM models
- Storing external agent conversation history in KMS beyond what is captured in `kms_acp_tool_calls`
- OAuth2 flows for external agent API key management (API keys stored in environment variables only)
- UI for managing external agent configuration (API only in M15; UI deferred to M16)
- Rate limiting or billing per external agent call
- External agent response caching
- Websocket or bidirectional-SSE transport for stdio adapters (unidirectional SSE relay only)
- Any non-ACP protocol for external agent communication (gRPC, raw REST)

## 4. Functional Requirements

| ID    | Priority | Requirement                                                                                                                                                                  |
|-------|----------|------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| FR-01 | Must     | `GET /api/v1/external-agents` returns the list of all configured external agents with their `agentId`, `type` (stdio/http), `enabled` state, and last health-check result    |
| FR-02 | Must     | `POST /api/v1/external-agents/test` accepts `{ agentId }` and performs a minimal round-trip to the agent (ping or capabilities call); returns latency and `healthy: boolean` |
| FR-03 | Must     | `ExternalAgentRegistry` in `kms-api/src/modules/acp/external-agent/` maps agent IDs to adapter config (type, command/URL, API key env-var, model, context window tokens)     |
| FR-04 | Must     | `StdioAcpAdapter` spawns the agent subprocess, communicates over stdin/stdout using the ACP wire format, and tears down the process when the session ends                    |
| FR-05 | Must     | `HttpAcpAdapter` connects to Anthropic API or any ACP-over-HTTP endpoint; handles retry with exponential back-off on 429 and 503 responses                                  |
| FR-06 | Must     | `ExternalAgentSessionManager` tracks subprocess PID or HTTP session for each active external agent session; cleans up on timeout or explicit cancel                          |
| FR-07 | Must     | `POST /api/v1/chat/runs` accepts an optional `externalAgentId` parameter; when present, generation is routed to the specified external agent instead of local Ollama          |
| FR-08 | Must     | Context Packer (`rag-service/app/services/context_packer.py`) converts RAG retrieval results (chunks + graph nodes) into ACP resource blocks                                 |
| FR-09 | Must     | Context Packer enforces per-agent token budget; excess chunks are dropped (hard truncation, lowest-score first) with a log warning                                           |
| FR-10 | Must     | Context Packer injects citation blocks: each resource block includes `source_id`, `file_name`, `chunk_index`, and `score` in its metadata                                   |
| FR-11 | Must     | Context Packer includes graph context (entity nodes + relationship edges) in a dedicated resource block when `kms_graph_expand` results are available                        |
| FR-12 | Must     | ACP Client (`kms-api`) sends packed context + user question to the external agent's ACP endpoint and streams the response back to the user via SSE                           |
| FR-13 | Must     | Citation annotations from the original RAG context are preserved in the SSE relay: the `done` event includes a `citations` array with chunk provenance                      |
| FR-14 | Must     | MCP Server exposes `GET /mcp/v1/tools` returning a JSON Schema manifest of available KMS tools (`kms_search`, `kms_retrieve`, `kms_graph_expand`)                           |
| FR-15 | Must     | MCP Server exposes `POST /mcp/v1/tools/call` accepting `{ toolName, input }`, dispatching to the KMS Tool Registry, and returning the result synchronously                  |
| FR-16 | Must     | All MCP endpoints require a valid JWT; missing or invalid JWT returns 401 KBEXT0001 (not KBACP0001 — separate domain)                                                        |
| FR-17 | Must     | `kms_ask_agent(agent_id, context, question)` tool sends a pre-formed context + question to an external agent and returns the agent's response text                           |
| FR-18 | Must     | `kms_delegate(agent_id, task, context)` tool delegates an entire task description with structured context to an external agent; intended for workflow step delegation         |
| FR-19 | Must     | WorkflowEngine recognises step type `external_agent` with fields `{ agentId, taskTemplate, contextSources }` and routes execution to the configured external agent           |
| FR-20 | Must     | `external_agent` workflow step passes all prior step outputs as context to the external agent, formatted by Context Packer                                                    |
| FR-21 | Must     | Feature flag `externalAgents.enabled: false` causes `POST /api/v1/chat/runs` with `externalAgentId` and `kms_ask_agent` / `kms_delegate` tool calls to return 503 KBEXT0001 |
| FR-22 | Must     | Feature flag `mcp.enabled: false` causes all `/mcp/v1/*` endpoints to return 503 KBEXT0009                                                                                  |
| FR-23 | Must     | Each external agent invocation records a row in `kms_acp_tool_calls` with `external_agent_id`, `context_tokens_sent`, `response_tokens`, `agent_model`, `duration_ms`        |
| FR-24 | Must     | `StdioAcpAdapter` subprocess startup must complete within 5 seconds; if the process does not emit an ACP capabilities response, it is killed and KBEXT0003 is returned       |
| FR-25 | Must     | External agent session timeout (configurable, default 120 s) terminates the subprocess or HTTP request and returns KBEXT0004                                                 |
| FR-26 | Should   | Configurable context truncation mode per agent: `hard` (drop lowest-score chunks) or `summarise` (use local LLM to compress the oldest chunks); default is `hard`            |
| FR-27 | Should   | `GET /api/v1/external-agents` includes a `lastHealthCheck` timestamp and `consecutiveFailures` counter for each agent                                                        |
| FR-28 | Should   | A background health-check job runs every 5 minutes against each enabled external agent; marks agent `unhealthy` after 3 consecutive failures                                 |
| FR-29 | Should   | `POST /api/v1/chat/runs` with an unhealthy external agent returns 503 KBEXT0012 rather than attempting the session                                                           |
| FR-30 | Could    | Context Packer exposes a `dry_run` mode that returns token counts and what would be truncated, without sending to the external agent; useful for debugging context budgets    |

## 5. Non-Functional Requirements

| Concern         | Requirement                                                                                                       |
|-----------------|-------------------------------------------------------------------------------------------------------------------|
| Performance     | Context packing p95 < 100 ms for up to 20 retrieved chunks + graph context (measured in rag-service)             |
| Performance     | External agent first-token latency p95 < 3 s from KMS receiving the request (excludes network to external agent) |
| Performance     | MCP tool call (`/mcp/v1/tools/call`) end-to-end p95 < 500 ms excluding downstream KMS tool latency               |
| Performance     | Stdio subprocess startup p95 < 2 s from `spawn()` call to first ACP capabilities frame received                  |
| Performance     | `GET /api/v1/external-agents` (list + cached health status) p95 < 50 ms                                          |
| Security        | External agent API keys must never appear in logs, SSE streams, or `kms_acp_tool_calls` records                  |
| Security        | API keys are read from environment variables only; they must not be stored in the database or config.json         |
| Security        | All MCP and external-agent endpoints require JWT; tool inputs validated against JSON Schema before dispatch        |
| Security        | Context payload sent to external agents must not include other users' data; session scoping enforced at Context Packer |
| Security        | Subprocess adapter must run subprocesses with restricted file-system access (no write permissions outside /tmp)   |
| Scalability     | `ExternalAgentSessionManager` must support at least 50 concurrent external agent sessions per kms-api instance    |
| Scalability     | Subprocess adapter must cap total spawned processes at `externalAgents.maxConcurrentSubprocesses` (default 10)    |
| Availability    | External agent unavailability must degrade gracefully: fallback to local Ollama if `externalAgents.fallbackToLocal: true` |
| Availability    | MCP server availability follows kms-api SLA (99.5% uptime); it does not depend on any external agent being healthy |
| Data retention  | External agent call records in `kms_acp_tool_calls` retained for 30 days (same policy as existing tool calls)    |
| Observability   | Every external agent call must emit an OTel span: `ext_agent.call` with attributes `agent_id`, `context_tokens`, `response_tokens`, `duration_ms` |
| Observability   | Subprocess adapter emits `ext_agent.process.start`, `ext_agent.process.ready`, `ext_agent.process.exit` OTel events |

## 6. Data Model Changes

No new tables are required. The existing `kms_acp_tool_calls` table is extended with four nullable columns to accommodate external agent call metadata.

```sql
-- Migration: 20260317000002_extend_acp_tool_calls_for_external_agents

ALTER TABLE kms_acp_tool_calls
  ADD COLUMN external_agent_id    TEXT,          -- e.g. "claude-api", "codex", "custom-acp"
  ADD COLUMN context_tokens_sent  INTEGER,       -- tokens in the context payload sent to the external agent
  ADD COLUMN response_tokens      INTEGER,       -- tokens in the external agent's response
  ADD COLUMN agent_model          TEXT;          -- model string reported by external agent e.g. "claude-sonnet-4-6"

CREATE INDEX idx_acp_tool_calls_external_agent_id
  ON kms_acp_tool_calls (external_agent_id)
  WHERE external_agent_id IS NOT NULL;
```

**Prisma schema changes** (`kms-api/prisma/schema.prisma`):

```prisma
model KmsAcpToolCall {
  // ... existing fields ...

  externalAgentId   String?  @map("external_agent_id")
  contextTokensSent Int?     @map("context_tokens_sent")
  responseTokens    Int?     @map("response_tokens")
  agentModel        String?  @map("agent_model")
}
```

No changes to `kms_acp_sessions`, `kms_files`, or any other table. The migration is additive-only; all new columns are nullable so existing rows are unaffected.

## 9. Decisions Required

| # | Question | Options | Recommendation | ADR |
|---|----------|---------|---------------|-----|
| 1 | **Subprocess vs HTTP for claude-agent-acp** — The stdio adapter spawns a subprocess per session (process management overhead, Docker container complexity) whereas direct HTTP to the Anthropic API is stateless and simpler. | (A) Stdio subprocess for claude-agent-acp; (B) Direct HTTP to Anthropic Messages API; (C) Support both, select per config | (B) for claude-api (direct Anthropic Messages API); stdio only for agents with no HTTP alternative (Kimi local, custom CLI agents). Reduces Docker complexity significantly. | ADR-0023 |
| 2 | **Context truncation: hard truncate vs summarise** — Hard truncation drops the lowest-scoring chunks when the budget is exceeded. Summarisation uses the local Ollama to compress older chunks, preserving more information but adding latency (100–500 ms extra). | (A) Hard truncate only; (B) Summarise mode as opt-in per agent; (C) Always summarise | (B) Hard truncate as default with `"truncationMode": "summarise"` as opt-in. Most queries stay under budget; summarisation overhead only for users who explicitly configure it. | ADR-0023 |
| 3 | **MCP vs ACP for Claude Code integration** — MCP is a tool-call protocol (synchronous request/response, no session state); ACP is a full agent session protocol (stateful, streaming). Claude Code natively supports MCP; ACP requires a wrapper. | (A) MCP server only; (B) ACP server only (Claude Code via ACP client adapter); (C) Both, with MCP as primary | (C) MCP as the primary interface for Claude Code (native support, simpler, no session state to manage). ACP remains the primary protocol for full-session external agents (Pattern 1). | ADR-0023 |
| 4 | **Process cap enforcement** — Uncapped subprocess spawning creates resource exhaustion risk. The cap can be enforced at the kms-api level (reject new sessions) or at the OS level (cgroup limits in Docker). | (A) Application-level cap with queue; (B) Reject with 503 when cap reached; (C) OS cgroup limits only | (B) Application-level cap with immediate 503 KBEXT0003 when `maxConcurrentSubprocesses` is reached. Simpler than a queue for the first iteration. | ADR-0023 |

### Rollout Sequence

1. Deploy M15 code with all flags at `false` — zero user impact; no new HTTP routes are reachable
2. Enable MCP server first: `mcp.enabled: true` — lower risk (read-only tools, no external calls). Run MCP integration suite against staging.
3. Enable single external agent for internal testing: `externalAgents.enabled: true` + `claudeApi.enabled: true` with a test API key. Run Pattern 1 E2E suite.
4. Validate audit logging: confirm `kms_acp_tool_calls` records populated with `external_agent_id`, `context_tokens_sent`, `response_tokens`.
5. Enable background health-check job; verify health-check results in `GET /api/v1/external-agents`.
6. Production rollout: enable one agent at a time, monitor p95 latencies, error rates (KBEXT* codes), and Anthropic API cost metrics for 48 h per agent.
7. Enable stdio-based agents (Codex, Gemini) last — highest operational complexity.

### Migration

1. Run Prisma migration `20260317000002_extend_acp_tool_calls_for_external_agents` — additive ALTER TABLE; no data backfill
2. Update `KmsAcpToolCall` Prisma model to include new nullable columns
3. No seed data required
4. No changes to any existing table structure; fully backwards-compatible

### Dependencies

| Dependency              | Required For                                           | Owner           |
|-------------------------|--------------------------------------------------------|-----------------|
| PRD-M13 (ACP Gateway)   | ACP tool registry, session infrastructure              | M13 complete    |
| PRD-M14 (Workflow Engine)| `external_agent` step type; `kms_delegate` tool       | M14 complete    |
| Redis (existing)        | External agent session state (reuses ACP session store) | Platform infra  |
| search-api (existing)   | `kms_search` tool via MCP                              | M13             |
| Qdrant (existing)       | `kms_retrieve` tool via MCP                            | M04             |
| Neo4j (existing)        | `kms_graph_expand` tool via MCP                        | M09             |
| ADR-0023                | Adapter pattern + truncation + process cap decisions   | Team decision   |
| ANTHROPIC_API_KEY       | claude-api and claude-code agents                      | Operator config |

### Rollback Plan

1. Set `externalAgents.enabled: false` — all external agent routes immediately return 503 KBEXT0001; no data loss
2. Set `mcp.enabled: false` — MCP endpoints return 503 KBEXT0009; does not affect any other KMS functionality
3. The ALTER TABLE migration is additive; rollback migration drops the four new nullable columns with no impact on existing rows
4. Any in-flight subprocess sessions are terminated by `ExternalAgentSessionManager.shutdown()` on kms-api restart; subprocesses exit cleanly
5. `kms_acp_tool_calls` rows with `external_agent_id` populated remain for audit purposes; they do not affect system operation
