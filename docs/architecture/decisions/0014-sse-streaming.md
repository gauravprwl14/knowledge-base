# 0014 — SSE over WebSocket for LLM Response Streaming

- **Status**: Accepted
- **Date**: 2026-03-30
- **Deciders**: Architecture Team
- **Tags**: [api, streaming, sse, rag]

## Context and Problem Statement

The RAG chat feature (`rag-service`) streams LLM token responses to the frontend as they are generated by the language model. The connection must push data continuously from server to client while the LLM produces output. Two transport options were evaluated: WebSocket (bidirectional, stateful) and Server-Sent Events (SSE, unidirectional, HTTP-native).

The same decision applies to the scan progress realtime updates feature (`kms-api` sources module).

## Decision Drivers

- LLM streaming is inherently unidirectional: server produces tokens, client consumes them
- Connection must survive standard reverse proxy setups (nginx) without special configuration
- Browser native support without additional client libraries
- Simple retry/reconnect semantics on connection drop
- Minimal infrastructure change to existing HTTP pipeline

## Considered Options

- Option A: Server-Sent Events (SSE) via `StreamingResponse` in FastAPI / `res.raw` in NestJS
- Option B: WebSocket (bidirectional, stateful upgrade)
- Option C: gRPC server streaming (binary, HTTP/2)
- Option D: HTTP long-polling (repeated requests with timeout)

## Decision Outcome

Chosen: **Option A — Server-Sent Events (SSE)**

SSE is a standard HTTP response with `Content-Type: text/event-stream`. It requires no protocol upgrade, works through standard nginx reverse proxy configurations without `proxy_pass` upgrade headers, and natively matches the unidirectional server-to-client data flow of LLM token streaming. The browser `EventSource` API handles automatic reconnection. FastAPI's `StreamingResponse` with an async generator is the idiomatic implementation.

### Consequences

**Good:**
- No nginx `Upgrade` header configuration required — works through existing proxy setup
- Browser `EventSource` API handles reconnect automatically with `Last-Event-ID` support
- HTTP/1.1 compatible — no HTTP/2 requirement
- FastAPI `StreamingResponse` with `async for token in llm_stream` is clean and idiomatic
- Connection closes naturally when the LLM finishes — no explicit close handshake
- Works correctly with JWT auth via `Authorization` header on initial connection

**Bad / Trade-offs:**
- SSE is text-only (UTF-8); binary payloads require base64 encoding (not needed for LLM tokens)
- SSE connections over HTTP/1.1 are subject to browser connection limits per domain (6 connections) — mitigated by HTTP/2
- No bidirectional channel — if the user wants to cancel a generation, a separate `DELETE /chat/:id` HTTP call is needed

## Pros and Cons of the Options

### Option A: SSE — CHOSEN

- ✅ Standard HTTP — no protocol upgrade
- ✅ Works through nginx without extra configuration
- ✅ Browser `EventSource` built-in; no JS library needed
- ✅ Unidirectional — matches LLM streaming and scan progress use cases exactly
- ✅ FastAPI `StreamingResponse` is idiomatic and well-documented
- ❌ Text-only protocol
- ❌ HTTP/1.1 connection limit per domain (mitigated by HTTP/2)
- ❌ No server-initiated close acknowledgement from client

### Option B: WebSocket

- ✅ Bidirectional — could support mid-stream cancellation or client messages
- ✅ Lower per-message framing overhead for many small messages
- ❌ Requires nginx `proxy_pass` with `Upgrade` and `Connection` headers — extra config
- ❌ Stateful connection — more complex server state management
- ❌ Bidirectional overhead not needed for one-way LLM streaming
- ❌ ADR-0033 already selected SSE-adjacent approach for file status; consistency favours SSE

### Option C: gRPC server streaming

- ✅ Binary protocol — efficient for large payloads
- ✅ HTTP/2 multiplexed — no connection limit issues
- ❌ Browser support requires gRPC-Web proxy (additional infrastructure)
- ❌ Protocol Buffers add schema overhead for a simple text token stream
- ❌ Frontend complexity: grpc-web library required

### Option D: HTTP long-polling

- ✅ Simplest server implementation — standard HTTP responses
- ✅ Works everywhere
- ❌ High latency between tokens (one round-trip per poll)
- ❌ Server must buffer tokens between polls — state management complexity
- ❌ Poor user experience for real-time token streaming
